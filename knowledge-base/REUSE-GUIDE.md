# WorkBuddy 教学视频模板复用方法

## 目标

以后制作第 2 集、第 3 集时，保留现有视觉系统、字幕样式、女声配置和动画结构，只替换内容、截图和当集实操录屏。这样每集画面统一，观众也能跟着真实界面操作。

## 一集视频的标准结构

建议保持 7—9 个场景：开场承诺、概念解释、准备条件、界面或步骤、演示任务、核心方法、安全提醒、验收与下集预告。第一集采用 8 个场景，总时长约 2 分 48 秒。

## 只替换这些内容

打开 `content/episode.json`：

- `title`、`subtitle`：本集主题与观众收益。
- `scenes[].title`：每页只讲一个结论。
- `scenes[].claim`：画面上最大的口语化解释。
- `scenes[].bullets`：3—4 个短要点。
- `scenes[].narration`：与本页画面对应的旁白。
- `scenes[].sourceImage`：需要展示的界面截图。
- `scenes[].demoVideo`：当集核心操作的 WorkBuddy 真实录屏。
- `accent`：在橙、蓝、薄荷绿、莓红、阳光黄中轮换。

不要直接改生成后的 `hyperframes/index.html`；它会由内容文件重新生成。

## 旁白规则

1. 一句话只表达一个动作或结论。
2. 第一次出现的术语马上用日常语言解释。
3. 画面先出现关键词，旁白再补充原因和注意事项。
4. 单场景控制在 15—25 秒。
5. 保留 `voice.id = female_question_reference_999`，并校验固定参考 WAV 的 SHA-256，即可延续“男女访谈式”中的同一位女声。
6. 使用 CosyVoice3 `inference_instruct2` 整段生成场景旁白，速度固定 `1.0`，保留真实停顿、呼吸和轻微句尾转折。
7. 每段生成同一组固定种子的至少 3 个候选，先校验相对参考音色相似度不低于 `0.60`，再对全片做连续选择；任意相邻两段音色相似度必须不低于 `0.86`。
8. 不允许对局部片段变速、变调或切掉句尾；只裁掉边缘静音，并保留开头 20—30ms、结尾 40ms 左右的自然余量。

## 画面规则

- 背景使用暖白色，避免长时间纯白刺眼。
- 标题用深蓝黑，重点用高饱和橙、蓝、绿、红、黄。
- 每页只保留一个视觉焦点，WorkBuddy 界面占满 1920×972 教学区，只叠加短标题、单句提示和一个定位框。
- 字幕使用独立的 108px 安全带，页面整体缩放到上方 972px；字幕不得覆盖 PPT 内容。
- 每个功能讲解页必须出现 WorkBuddy 官方界面截图，并用高亮框、编号或箭头标出操作位置。
- 每集必须录制本机 WorkBuddy 的核心操作过程，旁白边播放录屏边解释；静态截图只用于停顿讲控件或展示结果。
- 第 0 帧必须直接出现清晰界面和本集主题，禁止从黄色空白、纯色背景或延迟淡入开始。
- 禁止使用“左边大字占半屏、右边界面缩得看不清”的分栏版式。
- 转场以推入和柔焦为主，不使用连续闪烁或无意义炫技。

## 标准制作命令

```powershell
npm run validate
npm run build:audio
npm run build:slides
npm run build:hyperframes
Set-Location hyperframes
npm run check
npm run render -- --output ..\outputs\episode.mp4 --fps 30 --quality high
Set-Location ..
npm run package:engineering
```

## 每集交付标准

- MP4 可完整解码，1920×1080，30fps，包含声音。
- 女声与场景切换同步，字幕没有跨场残留。
- 女声参考文件哈希匹配，单段音色相似度不低于 0.60，相邻片段音色相似度不低于 0.86。
- PPT 可编辑、无溢出、无被裁切文字。
- 从第 0 帧开始每一页都有可读的 WorkBuddy 实际界面和明确操作定位，核心步骤包含真实录屏。
- 官方界面截图注明来源，不虚构产品能力。
- 高风险操作要明确限制范围、禁止删除或覆盖，并提醒人工验收。
- 运行 `npm test` 与 HyperFrames `check` 后再发布。

## 新一集命名建议

使用 `workbuddy-episode-02`、`workbuddy-episode-03` 这样的连续编号；视频、PPT、封面和内容文件使用相同编号，避免素材混淆。
